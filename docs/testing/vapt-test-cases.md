## Category 1 — Authentication & Session (TC-SEC-001 to TC-SEC-010)

---

### TC-SEC-001: OAuth State Parameter Tampering

**Category:** Auth Bypass  
**Severity:** Critical  
**Precondition:** Browser with DevTools open; ability to intercept and modify redirect URI parameters during OAuth flow.

**Steps:**
1. Navigate to `/login` and click "Sign in with Google".
2. Intercept the redirect to `https://accounts.google.com/o/oauth2/auth` before it completes (use a proxy such as Burp Suite or browser DevTools).
3. Note the `state` query parameter in the authorization URL generated by NextAuth.
4. Allow the OAuth flow to proceed to the Google consent page, then capture the callback URL on return to `/api/auth/callback/google`.
5. Modify the `state` parameter in the callback URL to an arbitrary value (e.g., `state=tampered123`).
6. Submit the modified callback URL.

**Expected Result:** NextAuth rejects the callback with an error (state mismatch); no session is created; user is not authenticated.

**Failure Indicator:** A session cookie is issued despite the modified state parameter, indicating CSRF protection on the OAuth flow is absent or bypassable.

**Sprint:** Sprint 1

---

### TC-SEC-002: Session Cookie Theft Simulation

**Category:** Session Hijacking  
**Severity:** Critical  
**Precondition:** Two browser profiles (Victim and Attacker). Victim is authenticated and holds a valid `__Secure-next-auth.session-token` (or `next-auth.session-token` in dev) cookie.

**Steps:**
1. In Victim browser, complete Google sign-in and confirm session is active at `/dashboard/calendar`.
2. Copy the full value of the session cookie from Victim's DevTools → Application → Cookies.
3. In Attacker browser (no session), open DevTools and manually set a cookie named `next-auth.session-token` (or `__Secure-next-auth.session-token`) with the copied value, targeting the DayFlow origin.
4. Navigate to `/dashboard/calendar` in the Attacker browser.

**Expected Result:** In production, the `__Secure-` prefix prevents the cookie from being set over non-HTTPS. In all environments, the session should be bound to the original request context where possible.

**Failure Indicator:** The Attacker browser gains full authenticated access to the Victim's dashboard using only the stolen cookie value, with no additional verification (e.g., IP binding or user-agent check).

**Sprint:** Sprint 1

---

### TC-SEC-003: Auth Bypass via Direct URL Access

**Category:** Auth Bypass  
**Severity:** High  
**Precondition:** Unauthenticated browser session (no cookies).

**Steps:**
1. Without signing in, directly navigate to `/dashboard/calendar` in the browser address bar.
2. Also attempt: `/dashboard`, `/dashboard/anything`, `/dashboard/../login`.
3. Observe the HTTP response status and final URL.

**Expected Result:** Middleware (`src/middleware.ts`, matcher: `["/dashboard/:path*"]`) intercepts all requests matching the pattern and redirects unauthenticated users to `/login` (or `/api/auth/signin`). A 307/302 redirect is returned; dashboard content is never served.

**Failure Indicator:** Dashboard page HTML is returned with a 200 status for any unauthenticated request, or middleware is bypassed via path manipulation (e.g., double-slash, URL encoding).

**Sprint:** Sprint 1

---

### TC-SEC-004: Session Fixation

**Category:** Session Fixation  
**Severity:** High  
**Precondition:** Ability to craft a URL or set a cookie before the victim authenticates.

**Steps:**
1. As an unauthenticated user, record the session identifier (e.g., CSRF token or any pre-auth cookie) before clicking "Sign in with Google".
2. Complete the OAuth flow and authenticate successfully.
3. Compare the session token value before and after authentication.

**Expected Result:** NextAuth issues a completely new session token after successful authentication. Any pre-authentication token or identifier is invalidated and not reused as the post-authentication session.

**Failure Indicator:** The session token value is identical before and after login, or a pre-set cookie value from an attacker-controlled source is accepted as a valid post-authentication session.

**Sprint:** Sprint 1

---

### TC-SEC-005: Expired Access Token Handling

**Category:** Token Security / Auth  
**Severity:** High  
**Precondition:** Authenticated session. Access to the JWT token expiry logic in `auth.ts` (line 52: `if (Date.now() < (token.accessTokenExpires as number) * 1000)`).

**Steps:**
1. Sign in successfully and note the time.
2. Using a test environment with a shortened token lifetime (or by manually setting `accessTokenExpires` to a past Unix timestamp in the JWT via a debug endpoint), simulate an expired access token.
3. Navigate to `/dashboard/calendar` and trigger a calendar data fetch via `GET /api/calendar/events`.
4. Observe whether the refresh token logic fires (`refreshAccessToken()`) or if a stale/expired token is used.

**Expected Result:** The JWT callback detects the expired token, calls `refreshAccessToken()`, and the API route receives a fresh, valid `accessToken`. No calendar API call is made with an expired token.

**Failure Indicator:** The expired `accessToken` is passed directly to `getTodaysEvents()` in `google-calendar.ts`, resulting in a 401 from Google but potentially leaking the token value in error logs; or the application silently fails without invalidating the session.

**Sprint:** Sprint 1

### TC-SEC-006: Refresh Token Abuse / Replay

**Category:** Token Security  
**Severity:** Critical  
**Precondition:** Knowledge of a `refreshToken` value (e.g., from a compromised JWT or server-side log).

**Steps:**
1. Obtain a valid `refreshToken` value (simulate via test environment with debug logging enabled).
2. Craft a manual POST request to `https://oauth2.googleapis.com/token` with the captured `refreshToken`, `GOOGLE_CLIENT_ID`, and `GOOGLE_CLIENT_SECRET`.
3. Observe if a new `access_token` is issued.
4. After the legitimate user revokes application access in Google Account settings, attempt the same refresh request again with the previously obtained `refreshToken`.

**Expected Result:** After revocation, Google rejects the refresh token. The `refreshAccessToken()` function returns `{ ...token, error: "RefreshAccessTokenError" }`. The application detects this error in the session callback and prompts re-authentication.

**Failure Indicator:** A revoked refresh token continues to yield new access tokens, or the application does not surface the `RefreshAccessTokenError` to force re-login, leaving the session in an indefinitely active but compromised state.

**Sprint:** Sprint 1

### TC-SEC-007: Sign-Out Completeness (Session Cleared)

**Category:** Session Management  
**Severity:** High  
**Precondition:** Authenticated user session at `/dashboard/calendar`.

**Steps:**
1. Sign in and confirm the session cookie is present.
2. Trigger sign-out via NextAuth's sign-out mechanism (`/api/auth/signout` or the Server Action equivalent).
3. After sign-out, inspect browser cookies for any remaining `next-auth.*` cookies.
4. Attempt to navigate directly to `/dashboard/calendar` using the browser's back button.
5. Attempt to replay the old session cookie (copied before sign-out) in a fresh request using DevTools or curl.

**Expected Result:** All `next-auth.*` session cookies are cleared (expired/deleted) on sign-out. The server-side session is invalidated. Replaying the old cookie returns a 401 or redirects to `/login`.

**Failure Indicator:** The old session cookie value remains valid server-side after sign-out, allowing access to protected routes without re-authentication. Browser back-button navigation reveals cached dashboard content without a server round-trip check.

**Sprint:** Sprint 1

### TC-SEC-008: NEXTAUTH_SECRET Weakness

**Category:** Cryptographic Weakness  
**Severity:** Critical  
**Precondition:** Access to the deployed `.env` file or environment variable configuration (simulates insider threat / misconfiguration audit).

**Steps:**
1. Check the `NEXTAUTH_SECRET` value in `.env` or the deployment platform's environment variable store.
2. Verify the secret is at least 32 random bytes (256 bits), generated via `openssl rand -base64 32` or equivalent.
3. Test if `NEXTAUTH_SECRET` is set to a weak, guessable, or default value (e.g., `"secret"`, `"nextauth"`, `"changeme"`, or the app name).
4. If weak, attempt to forge a JWT by signing a crafted payload with the known weak secret using a tool like `jwt_tool` or a custom script.

**Expected Result:** `NEXTAUTH_SECRET` is a cryptographically random value of sufficient length. A forged JWT is rejected by NextAuth.

**Failure Indicator:** `NEXTAUTH_SECRET` is a weak or predictable string. A JWT crafted and signed with the guessed secret is accepted by NextAuth, allowing full auth bypass and impersonation of any user.

**Sprint:** Sprint 1

---

### TC-SEC-009: Concurrent Session Handling

**Category:** Session Management  
**Severity:** Medium  
**Precondition:** Two separate browsers or browser profiles.

**Steps:**
1. Sign in to DayFlow on Browser A. Note the session token.
2. Sign in to DayFlow on Browser B using the same Google account.
3. Return to Browser A and attempt to access `/dashboard/calendar` and `GET /api/calendar/events`.
4. Sign out from Browser B.
5. Return to Browser A and again attempt to access protected resources.

**Expected Result:** By design, NextAuth v5 JWT sessions are stateless, so both sessions may remain valid concurrently. Document this as an accepted risk. Verify that sign-out on one device does not silently invalidate the other without user awareness (or that it does, if server-side session invalidation is implemented).

**Failure Indicator:** An unexpected error state where one session interferes with another, or where a sign-out on Browser B invalidates Browser A's token without any notification, causing confusing UX that could mask a security issue.

**Sprint:** Sprint 1

---

### TC-SEC-010: JWT Tampering

**Category:** Token Integrity  
**Severity:** Critical  
**Precondition:** Authenticated session; ability to read and modify the session cookie value.

**Steps:**
1. Sign in and copy the `next-auth.session-token` cookie value (this is a JWE/JWS token).
2. Decode the token using a JWT library or `jwt_tool` (note: NextAuth v5 encrypts JWTs by default using JWE with `NEXTAUTH_SECRET`).
3. Attempt to modify the payload (e.g., change the `sub` field to another user's ID, set `accessToken` to an arbitrary value, or change `error` field).
4. Re-sign/re-encrypt the token with either the known secret (if weak per TC-SEC-008) or an arbitrary key.
5. Set the modified token as the session cookie and attempt to access `/api/calendar/events`.

**Expected Result:** NextAuth rejects the tampered token because the JWE signature/MAC is invalid. A 401 Unauthorized response is returned and the user is redirected to sign in.

**Failure Indicator:** The tampered JWT is accepted, and the attacker gains access with a modified identity or injected `accessToken`, effectively bypassing authentication.

**Sprint:** Sprint 1

### TC-SEC-011: accessToken Exposure in Client Bundle

**Category:** Token Exposure  
**Severity:** Critical  
**Precondition:** Authenticated session; browser DevTools.

**Steps:**
1. Sign in and navigate to `/dashboard/calendar`.
2. Open DevTools → Sources and search all loaded JavaScript bundles for the string `accessToken`, `access_token`, or any Google OAuth token pattern (`ya29\.`).
3. Also inspect the `__NEXT_DATA__` script tag in the page HTML source for token values.
4. Review the `session` callback in `auth.ts` — confirm `accessToken` is intentionally excluded (the comment on line 57 states: "accessToken stays server-side only — NOT forwarded to client session").

**Expected Result:** No `accessToken` value appears in any client-side JavaScript bundle, `__NEXT_DATA__` payload, or rendered HTML. The `session` callback correctly omits `token.accessToken` from the session object returned to the client.

**Failure Indicator:** An `accessToken` (especially one matching the `ya29.` Google OAuth token pattern) is discoverable in client-side code or the `__NEXT_DATA__` hydration payload, exposing a live Google API credential to any user who views the page source.

**Sprint:** Sprint 1

---

### TC-SEC-012: accessToken in API Response Bodies

**Category:** Token Exposure  
**Severity:** High  
**Precondition:** Authenticated session.

**Steps:**
1. Sign in and open DevTools → Network tab.
2. Navigate to `/dashboard/calendar` to trigger a `GET /api/calendar/events` request.
3. Inspect the full JSON response body of the `/api/calendar/events` endpoint.
4. Check that the response only contains `{ events: [...], syncedAt: "..." }` and no token fields.
5. Also inspect any other API routes that may exist for token leakage.

**Expected Result:** The API response contains only calendar event data and `syncedAt`. No `accessToken`, `refreshToken`, `idToken`, or any credential field is present in the response body.

**Failure Indicator:** The response body includes any token field (e.g., `"accessToken": "ya29...."`) either intentionally (bad design) or accidentally (serialization of the full token object).

**Sprint:** Sprint 2

### TC-SEC-013: accessToken Visible in Browser Network Tab

**Category:** Token Exposure  
**Severity:** High  
**Precondition:** Authenticated session; DevTools Network tab.

**Steps:**
1. Sign in and open DevTools → Network → record all requests.
2. Trigger a calendar data refresh.
3. Inspect all request and response headers for `GET /api/calendar/events` and any preflight OPTIONS requests.
4. Search request headers (especially `Authorization`) for any bearer token being sent from the browser to DayFlow's own backend.
5. Confirm the `getToken()` call in `route.ts` extracts the token server-side from the encrypted session cookie — not from a client-supplied `Authorization` header.

**Expected Result:** No `Authorization: Bearer ya29....` header is sent from the browser. The token is extracted server-side from the encrypted session cookie by `getToken({ req, secret })`. The raw `accessToken` is never transmitted over a browser-visible channel.

**Failure Indicator:** A raw `accessToken` is visible as a request header, query parameter (e.g., `?token=ya29....`), or in a response header from any DayFlow endpoint.

**Sprint:** Sprint 2

### TC-SEC-014: Token Leakage via Referer Header

**Category:** Token Exposure  
**Severity:** Medium  
**Precondition:** Authenticated session; `Referrer-Policy` header is set to `strict-origin-when-cross-origin` per `next.config.ts`.

**Steps:**
1. Sign in and navigate to `/dashboard/calendar`.
2. If any page embeds a link or resource that includes a token as a URL query parameter (e.g., `/dashboard/calendar?token=ya29....`), click that link or load the resource.
3. Using a proxy, inspect the `Referer` header sent with requests to third-party origins (e.g., `lh3.googleusercontent.com` for profile images).
4. Verify the `Referrer-Policy: strict-origin-when-cross-origin` header from `next.config.ts` is present and effective.

**Expected Result:** The `Referrer-Policy` header prevents the full URL path (which could contain tokens) from being sent to cross-origin destinations. Only the origin is sent for cross-origin requests, and nothing for downgrades. No token appears in a `Referer` header to any third party.

**Failure Indicator:** A full URL containing token material in the query string is leaked via the `Referer` header to a cross-origin resource, or the `Referrer-Policy` header is absent from responses.

**Sprint:** Sprint 1

### TC-SEC-015: accessToken in Server Logs

**Category:** Token Exposure  
**Severity:** High  
**Precondition:** Access to server-side logs (application logs, platform logs, or a logging service). Test environment with logging enabled.

**Steps:**
1. Trigger a normal calendar data fetch via `GET /api/calendar/events`.
2. Also trigger an error scenario: call `GET /api/calendar/events` with an invalid/expired token to force the `catch` block in `route.ts` (line 19: `console.error("Calendar fetch error:", error)`).
3. Review server-side console output / log stream for the presence of token strings (`ya29.`, `access_token`, `refresh_token`).
4. Review the `refreshAccessToken()` function in `auth.ts` for any logging of token values in the `catch` block (line 25-27).

**Expected Result:** No `accessToken` or `refreshToken` value appears in any log line. Errors are logged with safe context (e.g., error message, HTTP status code) but never with raw credential values. The `catch` in `refreshAccessToken` does not log the token object.

**Failure Indicator:** A raw token value (especially `refreshToken`, which is long-lived) appears in a log line, which could be harvested from a logging platform, log file, or CI/CD output.

**Sprint:** Sprint 1

## Category 3 — API Security (TC-SEC-016 to TC-SEC-025)

---

### TC-SEC-016: Unauthenticated Access to /api/calendar/events

**Category:** Auth Bypass  
**Severity:** Critical  
**Precondition:** No session cookie. curl or Postman.

**Steps:**
1. Send a `GET` request to `https://<host>/api/calendar/events` without any session cookie or Authorization header:
   ```
   curl -i https://<host>/api/calendar/events
   ```
2. Also try with a malformed cookie: `Cookie: next-auth.session-token=invalid`.
3. Also try with an empty cookie: `Cookie: next-auth.session-token=`.

**Expected Result:** All requests without a valid session return `HTTP 401` with body `{"error":"Unauthorized"}`. The `getToken()` call returns `null` and the early-return guard on line 11 of `route.ts` fires before any Google API call is made.

**Failure Indicator:** Any request returns `HTTP 200` with calendar event data, or `HTTP 500` with an error that reveals internal details (Google API error message, stack trace) without first checking authentication.

**Sprint:** Sprint 2

---

### TC-SEC-017: Horizontal Privilege Escalation — Accessing Another User's Calendar

**Category:** Authorization / IDOR  
**Severity:** Critical  
**Precondition:** Two test Google accounts (User A and User B) both authenticated to DayFlow.

**Steps:**
1. Sign in as User A and obtain User A's session cookie.
2. Sign in as User B in a separate browser and obtain User B's session cookie.
3. Using User A's session cookie, attempt to request calendar data that belongs to User B by manipulating any user identifier in the request (e.g., query parameters like `?userId=<userB_id>` or `?calendarId=<userB_email>`).
4. Observe whether the API enforces that the `accessToken` in the JWT always maps to the authenticated user's own Google account.

**Expected Result:** The `/api/calendar/events` route uses the `accessToken` extracted from the requesting user's own encrypted JWT session cookie. There is no mechanism to specify a different user's calendar — `calendarId` is hardcoded to `"primary"` in `google-calendar.ts` (line 22). Any attempt to inject a different calendar ID via query param is ignored.

**Failure Indicator:** The API accepts a `calendarId` query parameter and uses it to fetch a different user's calendar data, or any query parameter allows cross-user data access.

**Sprint:** Sprint 2

### TC-SEC-018: Malformed Authorization Header

**Category:** Input Validation  
**Severity:** Medium  
**Precondition:** curl or Postman.

**Steps:**
1. Send `GET /api/calendar/events` with a malformed `Authorization` header:
   - `Authorization: Bearer `  (empty token)
   - `Authorization: Bearer null`
   - `Authorization: Bearer undefined`
   - `Authorization: Basic dXNlcjpwYXNz`  (Basic auth on a Bearer endpoint)
   - `Authorization: Bearer ya29.AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA` (syntactically valid but revoked)
2. In each case, observe the HTTP status code and response body.

**Expected Result:** `getToken()` in the route handler extracts the token from the encrypted session cookie, not from the `Authorization` header. Malformed `Authorization` headers are ignored. All requests without a valid session cookie return `HTTP 401 {"error":"Unauthorized"}`.

**Failure Indicator:** The route handler processes or attempts to use a token from the `Authorization` header, leading to unexpected errors, stack traces in the response, or (worst case) partial authentication.

**Sprint:** Sprint 2

### TC-SEC-019: Injection Attacks (SQL / NoSQL / Command)

**Category:** Injection  
**Severity:** Low (by architecture)  
**Precondition:** Authenticated session.

**Steps:**
1. Send `GET /api/calendar/events` with injection payloads in all available query parameters:
   - `?calendarId=' OR '1'='1`
   - `?timeMin='; DROP TABLE sessions; --`
   - `?q={"$where":"1==1"}`
2. Observe server response and any error messages.

**Expected Result:** The application does not use a SQL or NoSQL database for calendar data (data flows through Google's API). The `google-calendar.ts` client uses the Google API client library with typed parameters — no raw query construction. Injection payloads in unrecognized query parameters are ignored entirely.

**Failure Indicator:** An injection payload causes an unexpected application error, reveals internal query structure, or influences the data returned. Any indication that user input is concatenated into a query or command string.

**Sprint:** Sprint 2

### TC-SEC-020: Path Traversal in API Routes

**Category:** Path Traversal  
**Severity:** Medium  
**Precondition:** curl or Postman.

**Steps:**
1. Attempt path traversal variations on the calendar events endpoint:
   - `GET /api/calendar/events/../../../etc/passwd`
   - `GET /api/calendar/events%2F..%2F..%2Flib%2Fauth`
   - `GET /api/calendar/../../../lib/auth`
2. Attempt to access other internal Next.js routes via traversal:
   - `GET /api/calendar/events/../../../../_next/static/`

**Expected Result:** Next.js normalizes URL paths and routes requests based on the file system route structure. No traversal outside of the defined route handlers is possible. A 404 is returned for unrecognized paths.

**Failure Indicator:** Any traversal path returns a 200 response with file contents or sensitive data, or causes an unhandled server error revealing the filesystem layout.

**Sprint:** Sprint 2

### TC-SEC-021: HTTP Method Override on Read-Only Routes

**Category:** Input Validation / Method Override  
**Severity:** Medium  
**Precondition:** curl or Postman.

**Steps:**
1. The `/api/calendar/events/route.ts` exports only a `GET` handler. Attempt the following:
   - `POST /api/calendar/events` with an empty body
   - `DELETE /api/calendar/events`
   - `PATCH /api/calendar/events` with a JSON body
   - `PUT /api/calendar/events`
   - `GET /api/calendar/events` with header `X-HTTP-Method-Override: DELETE`
   - `POST /api/calendar/events` with header `X-HTTP-Method-Override: GET`

**Expected Result:** Next.js App Router returns `HTTP 405 Method Not Allowed` for any method other than `GET`. The `X-HTTP-Method-Override` header is not processed by Next.js by default and does not alter routing behavior.

**Failure Indicator:** A non-GET method returns a 200 response, triggers unintended behavior, or a method override header successfully changes the processing method (e.g., POST processed as GET).

**Sprint:** Sprint 2

### TC-SEC-022: Response Data Enumeration

**Category:** Information Disclosure  
**Severity:** Medium  
**Precondition:** Authenticated session.

**Steps:**
1. Call `GET /api/calendar/events` and capture the full JSON response.
2. Check the `CalendarEvent` interface fields returned: `id`, `title`, `start`, `end`, `attendeeCount`, `colorId`, `location`.
3. Compare against the raw Google Calendar API event object to identify any fields being inadvertently included beyond the mapped fields.
4. Check the `syncedAt` field to assess information exposure (server time disclosure).
5. Trigger an error (e.g., revoke Google access) and inspect the error response body for stack traces or internal details.

**Expected Result:** The response strictly maps to the `CalendarEvent` interface — no extra Google API response fields (e.g., `organizer.email`, `attendees[].email`, raw `htmlLink`, `conferenceData`) are exposed. Error responses return only `{"error": "Failed to fetch calendar"}` without stack traces.

**Failure Indicator:** The full Google API event object is forwarded to the client (over-sharing), or error responses include stack traces, file paths, or internal error messages.

**Sprint:** Sprint 2

### TC-SEC-023: Large Payload / DoS via API

**Category:** Rate Limiting / DoS  
**Severity:** High  
**Precondition:** Authenticated session; scripting capability.

**Steps:**
1. Send a high volume of rapid requests to `GET /api/calendar/events` (e.g., 100 requests per second for 10 seconds) using a load testing tool such as `ab`, `hey`, or `k6`.
2. Observe server response times and whether any rate limiting is applied.
3. Also send a single request with an excessively large `Cookie` header (e.g., 100KB of padding) and observe server behavior.
4. Attempt to send a `POST /api/calendar/events` with a very large JSON body (e.g., 10MB).

**Expected Result:** The application either rate-limits repeated requests from the same session/IP (returning `HTTP 429 Too Many Requests`) or is deployed behind infrastructure (CDN, API gateway) that handles this. Large payloads are rejected.

**Failure Indicator:** No rate limiting exists at the application or infrastructure level; the Google Calendar API quota is exhausted by a single malicious authenticated user; large payloads cause memory exhaustion or application crashes.

**Sprint:** Sprint 2

### TC-SEC-024: CORS Misconfiguration

**Category:** CORS  
**Severity:** High  
**Precondition:** curl or a browser-based cross-origin test page hosted at a different origin.

**Steps:**
1. Send a cross-origin request to `GET /api/calendar/events` with an `Origin` header from an attacker-controlled domain:
   ```
   curl -i -H "Origin: https://evil.com" https://<host>/api/calendar/events
   ```
2. Inspect the response for `Access-Control-Allow-Origin: *` or `Access-Control-Allow-Origin: https://evil.com`.
3. Send an OPTIONS preflight request:
   ```
   curl -i -X OPTIONS -H "Origin: https://evil.com" -H "Access-Control-Request-Method: GET" https://<host>/api/calendar/events
   ```
4. Check if `Access-Control-Allow-Credentials: true` is returned alongside a permissive `Access-Control-Allow-Origin`.

**Expected Result:** Next.js does not add permissive CORS headers by default. No `Access-Control-Allow-Origin` header is returned for cross-origin requests to the API route (or it is restricted to the application's own origin). A malicious cross-origin site cannot read API responses.

**Failure Indicator:** `Access-Control-Allow-Origin: *` is returned, or the origin is reflected back with `Access-Control-Allow-Credentials: true`, enabling a cross-origin attacker site to silently call the authenticated API on behalf of a victim user.

**Sprint:** Sprint 2

### TC-SEC-025: Rate Limiting Absence on Auth Endpoints

**Category:** Rate Limiting  
**Severity:** High  
**Precondition:** curl or load testing tool.

**Steps:**
1. Rapidly send `POST /api/auth/signin/google` requests (simulating brute-force or enumeration of the sign-in endpoint):
   ```
   for i in $(seq 1 50); do curl -s -o /dev/null -w "%{http_code}\n" -X POST https://<host>/api/auth/signin/google; done
   ```
2. Also rapidly request `GET /api/auth/session` (used to fetch session state):
   ```
   for i in $(seq 1 100); do curl -s -o /dev/null -w "%{http_code}\n" https://<host>/api/auth/session; done
   ```
3. Observe if any `429 Too Many Requests` responses are returned at any point.

**Expected Result:** The application or its hosting infrastructure rate-limits repeated requests to authentication endpoints, returning `HTTP 429` after a defined threshold.

**Failure Indicator:** All 50–100 requests return `200` or `302` with no rate-limit response, leaving the auth endpoints open to abuse (e.g., repeated OAuth initiation for token-grinding or session state enumeration).

**Sprint:** Sprint 1

## Category 4 — Security Headers (TC-SEC-026 to TC-SEC-032)

---

### TC-SEC-026: Content Security Policy Enforcement

**Category:** Header / XSS Defence  
**Severity:** High  
**Precondition:** Browser DevTools; authenticated session.

**Steps:**
1. Load `/dashboard/calendar` and open DevTools → Network → select the document response.
2. Inspect the `Content-Security-Policy` response header and compare to the policy defined in `next.config.ts`:

## Key Findings from Code Review

The following observations were made during the codebase review while authoring these test cases:

1. **TC-SEC-011 (accessToken — client exposure):** The `session` callback in `auth.ts` intentionally omits `accessToken` from the client session object. This is the correct pattern. Verify it holds under all code paths.

2. **TC-SEC-026 (CSP — unsafe-inline):** `script-src 'self' 'unsafe-inline'` in `next.config.ts` weakens the CSP significantly. Any injected inline script would execute. A future sprint should migrate to nonce-based or hash-based CSP.

3. **TC-SEC-031 (HSTS missing):** `next.config.ts` does not include a `Strict-Transport-Security` header. This should be added for production deployments.

4. **TC-SEC-037 (Open redirect — partially mitigated):** The login page Server Action uses a hardcoded `redirectTo: "/dashboard/calendar"`, which is safe. However, the `callbackUrl` query parameter accepted by NextAuth's `/api/auth/signin` route may still be exploitable and requires validation testing.

5. **TC-SEC-006 (Refresh token — no revocation detection):** The `refreshAccessToken()` catch block returns `{ ...token, error: "RefreshAccessTokenError" }` but the `session` callback only forwards `token.error` if present. Verify that the UI/client properly forces re-login when `session.error === "RefreshAccessTokenError"`.

6. **TC-SEC-003 (Middleware scope gap):** The middleware matcher covers only `/dashboard/:path*`. Routes such as `/api/calendar/events` are not covered by the edge middleware but are protected by in-route `getToken()` checks. Confirm no protected UI routes exist outside the matcher pattern.
